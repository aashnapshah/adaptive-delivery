Stage 02 · Recommendation generation

Three generation configs run the same case (outputs pre-baked — the configs only diverge with a live LLM). Each is shown as a full workup so cost and agreement are comparable.
Single LLM = paper's minimal-prompt control · MAI-DxO = five-physician virtual panel (Nori et al. 2025) · Ours = Quadruple-Aim advocates + coordinator.

Stage 02 · Compare vs what was actually ordered

Ground truth = the tests present in the case record. Agreement = Jaccard overlap of the ordered-test sets, plus a same-final-diagnosis flag.

Stage 03 · Appropriateness & harm scoring

A clinician grades recommendations against the rubric (RAND/UCLA appropriateness 1–9; Judge 1–5) and the console shows agreement with the automated scorer. Reused console embedded below.

Stage 04 · Delivery interface & clinician response

Deliver a recommendation via one of five actions; the simulated clinician accepts or dismisses with alert fatigue — p(accept) decays as the same alert repeats. Reused workspace embedded below.

Stage 05 · Adaptive delivery policy

Pre-computed over a 400-recommendation stream (one patient can't train a bandit). The policy chooses how to deliver each scored rec to realise clinical value without overloading the clinician.
Cumulative reward over the stream. Always-interrupt collapses under fatigue; never-alert wastes value; adaptive policies recover most of the value at far fewer interrupts.

Outcome metrics

The reward it maximises (per rec)

delivered and acted on → + clinical value (signed; a harmful test scores negative)

delivered (any action) → − attention cost of that action

suppress → − value withheld

Value comes from the appropriateness/harm score, not from whether the clinician clicked accept — it optimises appropriate uptake, not engagement. The policy sees only a noisy appropriateness estimate.
